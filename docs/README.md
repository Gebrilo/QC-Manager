# QC-Manager Documentation

Welcome to the QC-Manager knowledge base. This documentation is structured by audience and topic area.

## Structure

| Section | Audience | Content |
|---------|----------|---------|
| [business/](business/) | BAs, PMs, Stakeholders | Product overview, workflows, glossary |
| [technical/](technical/) | Engineers, Architects | Architecture, API, database, integrations |
| [features/](features/) | Everyone | Feature docs with acceptance criteria |
| [qa/](qa/) | QA Engineers, Testers | Test strategy, E2E scenarios, role tests |
| [operations/](operations/) | DevOps, SRE | Setup, deployment, monitoring, troubleshooting |
| [security/](security/) | Security Reviewers, Engineers | Auth, RBAC, secrets, audit |
| [internal/](internal/) | Team | ADRs, plans, inventory |
| [archive/](archive/) | Team | Deprecated and obsolete docs |

## Conventions

- Use [GitHub alerts](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax#alerts) for `[!NOTE]`, `[!IMPORTANT]`, `[!WARNING]`, `[!CAUTION]`
- Use Mermaid diagrams for architecture and workflows
- Use tables for roles, permissions, configuration, and rules
- Use `<details>` for collapsible technical details
- Status values: `Implemented`, `Partially Implemented`, `Planned`, `Deprecated`, `Needs Validation`
- Write relative links; verify after moves
