# Use a multi-field `value_maps` JSONB on tracker config; retire dead columns

`tuleap_sync_config.status_value_map` is structurally a single-field translation table — `{ "<tuleap_label>": "<qc_label>" }` — and the transform engine only consults it when `key === 'status'`. Every other enum field (`severity`, `priority`, `environment`, anything custom) passes through unmapped, which silently breaks bidirectional translation: outbound `resolveBindValue('severity', 'critical')` does an exact-case match against Tuleap's `"Critical"` and fails; inbound, the lowercase `BugFields.severity` Zod enum rejects Tuleap's titlecase value once we wire validation. We will replace the column with `value_maps JSONB DEFAULT '{}'` whose shape is `{ "<field_name>": { "<tuleap_label>": "<qc_label>", ... }, ... }`, keyed on the unified field name (`status`, `severity`, `priority`, `environment`, …). The transform engine renames `applyStatusMap` → `applyValueMap(fieldName, value, config)` and calls it for every field rather than only for `status`; fields with no entry in `value_maps[fieldName]` pass through unchanged. The migration backfills `value_maps = jsonb_build_object('status', status_value_map)` for existing rows, then drops `status_value_map`. We also drop the dead `field_mappings` and `status_mappings` columns (written by `POST /tuleap-webhook/config` but never read by any service or route — verify with a fresh grep over `apps/api/src/` before applying the migration). We rejected adding a parallel `value_maps` column alongside the old one (B): the codebase already has parallel examples (`field_mappings` / `artifact_fields`) where the dead column lingered indefinitely, and that's the entropy this ADR exists to stop.
