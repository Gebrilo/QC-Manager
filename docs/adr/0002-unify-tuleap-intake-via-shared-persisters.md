# Unify Tuleap webhook intake via shared persister services

The current `/tuleap-webhook/unified` route transforms incoming Tuleap payloads but does not persist them — the four legacy per-type routes (`/bug`, `/task`, `/user-story`, `/test-case`) still own all writes, including the special behaviours each subtype requires (bug source classification; task `reject`/`archive`/`delete` state machine; soft-delete revival; reporter→`owner_resource_id` resolution; `tuleap_task_history` archival). We will close the gap by extracting the per-type persistence logic into `apps/api/src/services/persisters/` modules (one per artifact subtype), exposing a `dispatchAction(unified, config)` entry point that both intake paths call. The unified route becomes the single front door: `fromTuleap()` → Zod validate → `dispatchAction()`. Per-type routes become thin compat shims that translate their legacy payload shape into a unified payload and call the same dispatcher — drift between the two intake paths becomes structurally impossible. Shims will be deleted in one PR once n8n is fully cut over and the per-type routes record zero traffic in `tuleap_webhook_log` for at least 7 consecutive days. We rejected (a) having `persistUnified()` make an internal HTTP hop to the per-type route — adds transport coupling and double-logs; (b) re-implementing persistence from scratch in the unified path — guarantees the kind of long-term drift already documented in the codebase between `n8n-workflows/` and `n8n/workflows/`.
