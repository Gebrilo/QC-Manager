# QC UUIDs are canonical for inter-artifact links

A linked **Test Case** can be identified three ways: the QC `test_case.id` UUID, the QC business key `test_case.test_case_id` (e.g. `"T-123"`), or the Tuleap `tuleap_artifact_id` integer (e.g. `140`). The codebase currently mixes all three: `bugs.linked_test_case_ids UUID[]` expects UUIDs but Tuleap webhooks send integers (which `sanitizeUuidArray` silently drops, losing the data — see CLAUDE.md), and the Phase 4 BugForm asks the user to type comma-separated strings like `"T-123, T-456"` which the outbound emitter then `Number()`s into `NaN` and posts to Tuleap as broken links. The same triple-identity exists for **User Story** and **Task**, all with the same failure modes. We will treat the QC UUID as the only canonical identity for cross-artifact references on the QC side: `linked_test_case_ids` stays `UUID[]`, `parent_story_id` becomes `UUID FK → user_stories.id`, etc. Frontend forms drop free-text inputs and use dropdown pickers backed by QC's existing list endpoints — the user picks a Test Case (or User Story) by display name and the form sends its UUID. Inbound webhooks resolve `tuleap_artifact_id → id` via a lookup against the relevant table before storing; if the linked artifact hasn't been ingested yet (race), the integer is stashed on a `pending_links JSONB` column and resolved on the next webhook for any artifact in the matching tracker. Outbound emitters resolve `id → tuleap_artifact_id` before posting to Tuleap, so the wire format Tuleap sees is always its native integer artifact IDs. We rejected storing Tuleap integer IDs on the QC side (B): no FK safety, two identities for the same row in the codebase, and the lookup happens either way (just at a different boundary). We rejected free-text business keys (C): typing `"T-123"` by hand is the bug we're fixing, not a target end state.
