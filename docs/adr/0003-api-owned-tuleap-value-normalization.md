# API owns Tuleap value normalization; n8n stays a thin forwarder

The current `tuleap-unified-sync.json` workflow normalizes Tuleap artifact values (extracts a key for each field, picks a value from `value` / `bind_value_ids` / `links` / `content`) and the `/tuleap-webhook/unified` route then re-runs the same logic on the same data. Both implementations have the same two bugs: the field-key preference order picks Tuleap's human `label` over its short `name`, so `BASE_FIELD_MAPPINGS` lookups silently miss; and select-box (`sb`/`rb`/`msb`) values are forwarded as raw `bind_value_ids` integers instead of being resolved against the tracker's field schema, so `status_value_map` enum mapping never fires. We will move all normalization to the API in a new `apps/api/src/services/tuleapValueNormalizer.js` that takes `(artifactValues, trackerId)` and returns a flat `{ [fieldName]: resolvedValue }` keyed on Tuleap's `name` (falling back to `label` only as a last resort), resolving every `bind_value_ids` array to the corresponding label(s) via the existing `tuleapFieldRegistry` (5-minute cache). The n8n workflow's `Normalize Payload` node shrinks to the bare minimum — flatten `tracker_id` / `project_id` / `action` / `artifact_id` to the top level, forward `artifact.values` raw, drop `normalized_values` entirely. We rejected putting normalization in n8n (cleaner for API but contradicts the "n8n is thin" principle and would mean a second Tuleap client lives in inline JS); we rejected duplicating the normalizer in both places (the codebase already shows one drift case between `n8n-workflows/` and `n8n/workflows/`).
