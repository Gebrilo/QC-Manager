# Unify outbound Tuleap routes via a shared emitter

The current `apps/api/src/routes/tuleapArtifacts.js` `POST /:type` handler has three coexisting code paths — a unified-payload path, a config-driven synthetic-unified path that uses a brittle camelCase→snake_case converter (`BAAuthor` → `b_a_author`), and a legacy fallback that calls the hardcoded `tuleapPayloadBuilder.js`. The `PATCH /:id` and `DELETE /:id` routes ignore unified payloads entirely, which is why `tuleapApi.updateUnified()` returns 400 today (a known gap from the unified Phase 1 status doc). We will mirror ADR 0002 on the outbound side: extract the per-type emission logic into `apps/api/src/services/emitters/{bug,task,user_story,test_case}.js` (or a single `services/tuleapEmitter.js` if the per-type variation is small) exposing one `emitToTuleap(unified, config, mode)` entry point where mode ∈ `create | update | delete`, and rewrite all three outbound routes to call it. The legacy `tuleapPayloadBuilder.js` and its tests are deleted — the transform engine plus `tuleapFieldRegistry` already cover what the builders did, and keeping them duplicates `BASE_FIELD_MAPPINGS` in two places, which is exactly the drift risk ADR 0002 already cited. Frontend forms must send the unified payload (which they already do after Phase 4); we accept no compat shape on the outbound API surface — the camelCase synthesizer in Path B was silent corruption and we'd rather fail loud on a malformed payload than silently mangle field names.
