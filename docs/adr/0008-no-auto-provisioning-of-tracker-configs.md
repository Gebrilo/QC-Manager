# `/unified` does not auto-provision missing tracker configs

The legacy `/tuleap-webhook/task` route auto-provisions a QC project and a `tuleap_sync_config` row when neither exists, hardcoding `tracker_type: 'task'` — which is already a known bug for non-task trackers and the kind of silent misconfiguration that's hard to detect after the fact (a Bug tracker auto-mapped as `tracker_type='task'` will accept webhooks and silently route artifacts to the wrong QC table). With ADR 0002 collapsing all four artifact types onto `/tuleap-webhook/unified`, we will *not* port that behaviour: `/unified` returns `404 Unconfigured` (with the tracker ID and project info in the response body) when no `tuleap_sync_config` row exists, and the operator must explicitly create the mapping via the Phase 2 `/settings/tuleap` admin UI. We rejected smart type detection from Tuleap's `tracker.item_name` (A) because admins routinely rename trackers, localize labels, or use custom names — silent misrouting is the failure mode that auto-provisioning was already producing. We rejected an audit-table soft-provision (C) as too much new schema for a workflow run a few times per year. Existing auto-provisioned rows are not migrated; operators fix wrong `tracker_type` values via the admin UI as they're noticed. The dead `provisionTuleapProject()` helper and the `tuleap_project_id`-based fallback in the unified route are removed — the only path to a config row is now the explicit one.
