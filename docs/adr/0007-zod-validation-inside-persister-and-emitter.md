# Validate unified payloads inside the persister and emitter, with separate strict and partial schemas

`apps/api/src/schemas/tuleapUnified.js` defines `UnifiedPayloadSchema` but neither the unified webhook nor the outbound route imports it — validation is currently dead code. We will run validation *inside* `dispatchAction(unified, config)` and `emitToTuleap(unified, config, mode)` rather than as Express middleware, because the per-type webhook shims (ADR 0002) translate legacy payload shapes into unified payloads inside the route handler — middleware would run before that translation and reject every legacy webhook. Co-locating validation with the consumer also keeps it close to the code that depends on it and makes the persister/emitter independently testable. We will use two schemas: the existing strict `UnifiedPayloadSchema` for inbound and outbound-create (Tuleap webhooks always carry a full snapshot; create forms gather all required fields before submit), plus a `UnifiedPatchSchema` derived via `deepPartial()` with the discriminator (`artifact_type`) and `project_id` re-required for outbound-update so PATCH callers can send a field diff. `emitToTuleap` picks the schema based on `mode`; delete needs no payload validation beyond the URL artifact ID.
